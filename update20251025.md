# 按需排除关联 with 参数增强设计（update20251025）

## 背景与目标
- 现状：调用 `with()` 会对实体上所有带 `@RelationModel` 的字段做预加载，且可递归多级。
- 诉求：查询时临时禁用某些关联。例如传入 `A::x`，本次查询不加载实体 A 中字段 x 的关联数据（包括 count 场景），其余关联照常加载。
- 约束：
  - 不影响 `hasWhere` 行为（whereHas 反向筛主表）。
  - 默认行为完全兼容（不配置时与当前一致）。

## 总体方案
- 在运行期“过滤执行的关联”而非“修改注解或移除元数据”。
- 新增“排除规则 exclude”的配置入口，通过链式 API 配置到 `ExecuteWithSelectService` 实例上。
- `HasRelationService.relationModel(...)` 在遍历每个关联字段时，根据“当前类 + 递归路径”判断是否跳过；未命中则按现有逻辑查询并回填；递归时沿用相同规则。
- `hasWhere` 依赖元数据做子查询 SQL 组装，排除规则不作用于 `hasWhere`。

## API 设计
- 排除式（首版目标，满足当前需求）
  - `BaseHasManyMapper` 新增：
    - `default ExecuteWithSelectService<T, V> withExcept(String... rules)`
      - 等价 `with().exclude(rules)`
  - `ExecuteWithSelectService` 新增：
    - `public ExecuteWithSelectService<T, C> exclude(String... rules)`
      - 解析并保存排除规则（去重、规范化）
- 规则语法
  - 形式：`Owner::fieldPath` 或 `fieldPath`
    - `Owner`：类名（SimpleName），大小写不敏感；仅在根层启用时可省略。
    - `fieldPath`：以 `.` 连接的路径，代表递归层级，如 `posts.comments`。
  - 匹配语义：
    - 根层字段：`"posts"` 与 `"User::posts"` 均可匹配 `User` 的根层 `posts`。
    - 递归字段：`"User::posts.comments"` 匹配从 `User` 出发的第二级 `comments`。
  - 规则未命中：忽略并输出 debug 日志，不中断主流程。

## 行为说明
- 命中排除规则的字段：不做关联 SQL、不做结果回填；`count=true` 字段同样跳过统计。
- 未命中：沿用现有逻辑。
- `deep` 与排除：递归深度仍生效；排除只影响命中路径，不影响同级/其他分支。
- `hasWhere`：不受排除影响，仍可对被排除的关联字段做 whereHas 过滤（因为排除只作用于 with 预加载函数，不修改关系元数据）。

```java
package io.github.skyleew.relationmapping.filter;

import lombok.Value;
import java.util.List;

@Value
public class ParsedRule {
    String ownerSimpleName;       // 可为 null，表示未指定 owner（仅在根层匹配）
    List<String> pathSegments;    // 例如 ["posts", "comments"]
}
```

```java
// src/main/java/org/jieyun/relationmapping/filter/RelationSelector.java
package io.github.skyleew.relationmapping.filter;

import cn.hutool.core.util.StrUtil;
import java.util.*;
import java.util.stream.Collectors;

public class RelationSelector implements RelationFilter {
    private final Set<ParsedRule> excludes;

    public RelationSelector(Collection<String> rules) {
        this.excludes = normalize(rules);
    }

    @Override
    public boolean skip(Class<?> ownerClass, String fieldPath, boolean isRootLevel) {
        String owner = ownerClass.getSimpleName();
        String normPath = normalizePath(fieldPath);
        for (ParsedRule r : excludes) {
            boolean ownerMatch = (r.getOwnerSimpleName() == null)
                ? isRootLevel
                : r.getOwnerSimpleName().equalsIgnoreCase(owner);
            if (!ownerMatch) continue;
            if (join(r.getPathSegments()).equalsIgnoreCase(normPath)) return true;
        }
        return false;
    }

    private static Set<ParsedRule> normalize(Collection<String> rules) {
        if (rules == null) return Collections.emptySet();
        Set<String> seen = new HashSet<>();
        Set<ParsedRule> out = new LinkedHashSet<>();
        for (String raw : rules) {
            if (cn.hutool.core.util.StrUtil.isBlank(raw)) continue;
            String s = raw.trim();
            String owner = null;
            String path = s;
            int idx = s.indexOf("::");
            if (idx > 0) {
                owner = s.substring(0, idx).trim();
                path = s.substring(idx + 2).trim();
            }
            String norm = (owner == null ? "" : owner.toLowerCase() + "::") + normalizePath(path).toLowerCase();
            if (!seen.add(norm)) continue;
            List<String> segs = Arrays.stream(path.split("\\.")).map(String::trim).filter(cn.hutool.core.util.StrUtil::isNotBlank).toList();
            if (segs.isEmpty()) continue;
            out.add(new ParsedRule(owner, segs));
        }
        return out;
    }

    private static String normalizePath(String p) {
        if (p == null) return "";
        return Arrays.stream(p.split("\\.")).map(String::trim).filter(cn.hutool.core.util.StrUtil::isNotBlank).collect(java.util.stream.Collectors.joining("."));
    }

    private static String join(List<String> segs) {
        return String.join(".", segs);
    }
}
```

## 代码改动点与实现步骤

### 1) BaseHasManyMapper 增加便捷方法
- 文件：`src/main/java/org/jieyun/relationmapping/mapper/BaseHasManyMapper.java`
- 变更：新增 `withExcept`，委托到 `with().exclude(...)`

示例补丁（供实现时参考）：
```java
default ExecuteWithSelectService<T, V> withExcept(String... rules) {
    return with().exclude(rules);
}
```

### 2) ExecuteWithSelectService 支持 exclude 规则
- 文件：`src/main/java/org/jieyun/relationmapping/service/ExecuteWithSelectService.java`
- 变更点：
  - 新增字段：`private RelationFilter relationFilter;`
  - 新增方法：`public ExecuteWithSelectService<T, C> exclude(String... rules)` 创建 `RelationSelector` 并赋值给 `relationFilter`。
  - 在 `sqlSelectList(...)`、`sqlSelectOne(...)` 分支中调用 `hasRelationService.relationModel(...)` 时，若 `relationFilter != null` 则使用重载版本传入过滤器，否则沿用原方法（保持兼容）。

示例补丁（关键片段）：
```java
import io.github.skyleew.relationmapping.filter.RelationFilter;
import io.github.skyleew.relationmapping.filter.RelationSelector;

private RelationFilter relationFilter;

public ExecuteWithSelectService<T, C> exclude(String... rules) {
    this.relationFilter = new RelationSelector(Arrays.asList(rules));
    return this;
}

// 原调用：
// hasRelationService.relationModel(this.entityClass, this.relationModelStructureMap, t, this.deep);
// 新调用（存在过滤器时）：
if (this.relationFilter != null) {
    hasRelationService.relationModel(this.entityClass, this.relationModelStructureMap, t, this.deep, this.relationFilter, "");
} else {
    hasRelationService.relationModel(this.entityClass, this.relationModelStructureMap, t, this.deep);
}
```

### 3) HasRelationService 增加带过滤器的重载
- 文件：`src/main/java/org/jieyun/relationmapping/service/HasRelationService.java`
- 现有方法：
  - `public void relationModel(Class<?> selfModelEntityClass, Map<String, RelationModelStructure> tableStructureMap, Object result, int deep)`
- 变更与新增：
  - 新增重载：`public void relationModel(Class<?> owner, Map<String, RelationModelStructure> map, Object result, int deep, RelationFilter filter, String parentPath)`
  - 原方法可作为兼容入口，内部 `return relationModel(selfModelEntityClass, tableStructureMap, result, deep, null, "");`
  - 在遍历 `map.keySet()` 时，计算当前路径 `currentPath = parentPath.isEmpty() ? key : parentPath + "." + key`；调用 `filter.skip(owner, currentPath, parentPath.isEmpty())` 判定是否跳过。命中则 `continue`。
  - 执行查询与回填逻辑保持不变；递归调用时传入 `nextOwnerClass`、`nextLevelRelations`、`deep - 1`、相同 `filter`、`currentPath`。

伪代码（关键逻辑）：
```java
public void relationModel(Class<?> owner, Map<String, RelationModelStructure> map, Object result, int deep, RelationFilter filter, String parentPath) {
    if (owner == null || result == null || deep <= 0) return;
    for (String key : map.keySet()) {
        String currentPath = parentPath == null || parentPath.isEmpty() ? key : parentPath + "." + key;
        if (filter != null && filter.skip(owner, currentPath, parentPath == null || parentPath.isEmpty())) {
            continue; // 跳过本级关联（含 count）
        }
        RelationModelStructure table = map.get(key);
        getResultRelationIds(result, table);
        if (table.getSelfTableStructure().getIds() == null || table.getSelfTableStructure().getIds().isEmpty()) continue;

        if (table.isCount()) {
            Map<String, Long> countResult = relationSqlService.getCountResult(table);
            if (countResult != null && !countResult.isEmpty()) {
                resultCountMerge(result, countResult, table);
            }
            continue;
        }

        List<Object> relationResult = relationSqlService.getResult(table);
        if (relationResult == null || relationResult.isEmpty()) continue;

        List<Object> casted = resultRelationMerge(result, relationResult, table);
        if (casted.isEmpty()) continue;

        // 递归执行下一层
        Class<?> nextOwner = table.getRelationTableStructure().getReflectClass();
        Map<String, RelationModelStructure> nextMap = RelationUtil.checkRelation(nextOwner);
        if (!nextMap.isEmpty()) {
            relationModel(nextOwner, nextMap, casted, deep - 1, filter, currentPath);
        }
    }
}
```

### 4) RelationUtil/RelationSqlService/Domain 不需要修改
- 注解扫描、元数据构建、SQL 组装/映射流程无需变更。
- 保持 `hasWhere` 与 `with` 的职责边界清晰：排除规则仅约束 with 预加载阶段。

## 用法示例
```java
@Autowired UserMapper userMapper;

// 1) 排除根层 posts 关联
List<User> users = userMapper.withExcept("posts").selectList(new QueryWrapper<>());

// 2) 指定 owner 排除根层 posts（等价）
List<User> users2 = userMapper.withExcept("User::posts").selectList(new QueryWrapper<>());

// 3) 排除嵌套 posts.comments（深度>1 才有意义）
List<User> users3 = userMapper.with(3).exclude("User::posts.comments").selectList(new QueryWrapper<>());

// 4) 与 hasWhere 并存：不加载 posts，但仍基于 posts 做过滤
List<User> users4 = userMapper
    .withExcept("posts")
    .hasWhere("posts", qw -> qw.eq("status", 1))
    .selectList(new QueryWrapper<>());
```

## 测试用例建议
- 根层排除（列表/单对象/统计 count）
- 递归排除（只影响命中路径，不影响同级分支）
- 与 `deep` 交互（深度不足时，嵌套排除不生效是预期）
- 与 `hasWhere` 并存（排除 with，但 whereHas 能正常筛主表）
- 不存在的规则（忽略、日志调试）
- 规则大小写与去重

## 兼容性与性能
- 兼容性：无破坏性更改；不配置 `exclude` 时行为与当前完全一致。
- 性能：排除可减少子表 SQL 次数与结果回填成本；规则匹配为 O(关联字段数)，开销可忽略。

## 风险与注意点
- 规则匹配仅做“完整路径匹配”，暂不支持通配符；若需要可后续增强（如 `User::posts.*`）。
- 解析 Owner 名使用 SimpleName，避免与包名耦合；若用户传入全限定名，可在后续版本支持。
- 被排除字段如果在 VO/序列化时强依赖（非空假定），需在业务侧注意判空处理。

## 验收标准（验收点）
- 不配置 `exclude`：行为与当前一致（回归通过）。
- `withExcept("posts")`：根层 `posts` 未加载，其他关联保持。
- `withExcept("User::posts.comments")`：仅跳过 `comments` 分支；`posts` 自身仍加载。
- 与 `hasWhere` 并存：可对 `posts` 做 whereHas，同时 `with` 不回填 `posts` 数据。
- 命中 `count=true` 的字段：统计不执行，结果为默认值（不写入或保持 null）。
- 规则大小写与重复：均能正确处理。

## 后续扩展（可选）
- 仅包含模式：`withOnly(String... rules)`，与 `exclude` 互斥。
- 通配符匹配：如 `User::posts.*`、`*.comments`。
- 全限定类名支持与包级过滤。
